\begin{frame}{Motivation}
\begin{itemize}
    \item With a defined search space, we need a strategy to \alert{navigate it efficiently}
    \item Even a compact search space can contain \textbf{thousands} of valid architectures - making the search strategy important
    \item The choice of search strategy directly impacts \alert{which architectures are found} and \alert{how much compute is spent}
\end{itemize}
\pause
\vspace{0.5em}
\begin{colorblock}[black]{sintefgrey}{Key Question}
Given a fixed evaluation budget, which search strategy finds the best-performing architecture — and why does it outperform simply sampling at random?
\end{colorblock}
\end{frame}

\begin{center}

The search strategy aims to find the optimal architecture.
There are two different categories commonly used in NAS:
\begin{itemize}
    \item Black-box optimization techniques
    \item One-shot techniques
\end{itemize}

\begin{frame}{Basic and Black-Box Optimization Techniques}
\begin{itemize}
    \item Basic algorithms
        \begin{itemize}
            \item Random search
            \item Local search
        \end{itemize}
    \item Bayesian optimization 
    \item Evolution algorithm
\end{itemize}
\begin{itemize}
    \item Widely used due to strong performance
    \item Tend to use more computational resources than one-shot techniques
\end{itemize}
\end{frame}

\begin{frame}{Basic Algorithms}
Random search:
\begin{itemize}
    \item Select architectures randomly from search space and fully train them
    \item Architecture with the best score is the output
\end{itemize}
\pause
    $\leadsto$ Strong baseline in constrained search spaces

% Pros:
% \begin{itemize}
%     \item Performs well on highly engineered search spaces with a high amount of high scoring architectures
% \end{itemize}
% Cons:
% \begin{itemize}
%     \item Not the best performance on large, diverse search spaces
% \end{itemize}
\end{frame}

\begin{frame}{Basic Algorithms}
Local search:
\begin{itemize}
    \item Iteratively trains and evaluates all neighbors of the best architecture found so far
    \item Neighborhood: All architectures that differ by one operation or edge
\end{itemize}
\pause
$\rightsquigarrow$  Strong baseline based on a step-by-step, neighborhood-based search that can be viewed as an mimic of human architectural development, in which small, locally optimal changes are mostly made
\end{frame}

\begin{frame}{Evolutionary and Genetic Algorithms}
Evolutionary algorithm:
\begin{itemize}
    \item Iteratively updating a population of architectures
    \item In each step, one (or more) parent architecture (mostly high-scoring architecture) in the population is sampled, combined, and mutated to create new children architectures
    \item Children are trained and added to the population and replace individuals (with worse scores)
\end{itemize}

\begin{frame}{Evolutionary and Genetic Algorithms}
Methods differ in several steps:
\pause
\begin{itemize}
    \item Initial population sampling 
    \begin{enumerate}
        \item Using trivial architectures
        \item Random sampling
        \item Hand-picked high-scoring architectures
    \end{enumerate}
    \pause
    \item Parent selection 
    \begin{enumerate}
        \item Tournament selection, i.e., selecting best architecture(s) out of random population
        \item Random sampling weighted by fitness
        \item Choosing current best architecture(s) as parents 
        \item Trade off exploration vs exploitation the best region found so far
    \end{enumerate}
    \pause
    \item Child generation 
    \begin{enumerate}
        \item Mutation: combine two networks, by e.g.\ adding or removing a layer, altering the hyperparameters of a layer, adding skip connections
        \item Crossover: combine two networks $\rightsquigarrow$ \textbf{permutation problem}: different graph encodings can represent equivalent network which can result in dysfunctional networks
    \end{enumerate}
\end{itemize}
\end{frame}

\begin{frame}{Regularized/ Aging Evolution~\liturl{Real et al. 2019}{https://arxiv.org/pdf/1802.01548}}
Regularized evolution is the most successful evolutionary algorithm method in NAS due to maintaining population diversity and preventing premature convergence
\pause
\begin{itemize}
    \item Search for blocks (cells) with 5 choices (applied on the NASNet search space~\liturl{Zoph et al. 2018}{https://openaccess.thecvf.com/content_cvpr_2018/papers/Zoph_Learning_Transferable_Architectures_CVPR_2018_paper.pdf}): 
        \begin{itemize}
        \item Choose two previous feature maps
        \item Choose operations for both (convolutions, pooling, etc.)
        \item Choose operation to merge (add or concat)
    \end{itemize}
    \only<2-3>{
    \begin{columns}
        \begin{column}{0.6\textwidth}
        \centering
        \includegraphics[width=0.85\linewidth]{07_NAS/images/03/Figure-2.pdf}

    \pause
    \item Evolution simply tackles this as a \textit{hyperparameter optimization} (HPO) problem with $2 \times 5$ variables (normal and reduction cell)
    \item Worst/oldest solution is dropped from the population
\end{itemize}
\end{frame}


\begin{frame}{Regularized/ Aging Evolution~\liturl{Real et al. 2019}{https://arxiv.org/pdf/1802.01548}}
\begin{itemize}

\begin{frame}{Bayesian Optimization for NAS}
Bayesian optimization (BO) is a powerful tool for optimizing expensive functions
\pause
\begin{itemize}
    \item Two key components
    \begin{enumerate}
        \item Probabilistic surrogate
        \item Acquisition function
    \end{enumerate}
    \pause
    \item Iterative algorithm
    \begin{enumerate}
        \item Selecting architecture maximizing the acquisition function (computing  using the surrogate)
        \item Training the architecture
        \item Retraining the surrogate
    \end{enumerate}
\end{itemize}
\end{frame}

\begin{frame}{Summary} 

Most important insights from this video:

\begin{enumerate}
    \item Basic algorithms perform well on highly engineered search spaces but face issues in larger diverse ones
    \item Black-box optimization methods are widely used due to their strong performance
    \begin{itemize}
        \item They can be used in joint optimization with other hyperparameters
        \item They allow for easier adaptation to new problems
        \item Can be combined with different performance estimation strategies
    \end{itemize}
    % \item Bayesian optimization tends to use more computational resources than one-shot techniques, but is very competitive against other approaches
\end{enumerate}
