\begin{frame}[c]{Why Do We Need Bayesian Optimization?}

Many optimization problems in AutoML (e.g., HPO or NAS) share three features:
\begin{itemize}
    \item The objective function is a \textbf{black box} -- no analytic form, no gradients
    \item Each evaluation is \textbf{expensive} -- training or resampling an ML model on a dataset can take minutes, hours, or days
    \item Evaluations are often \textbf{noisy} -- due to stochastic training, resampling, or measurement noise
\end{itemize}
\vspace{0.3cm}
\begin{center}
\textbf{Classic optimization methods do not fit well here}
\end{center}
\end{frame}


\begin{frame}{HPO Is a Black-Box Optimization Problem}
\begin{itemize}
    \item Finding a good hyperparameter configuration $\conf^\ast$ means solving
    $$\conf^\ast \in \argmin_{\conf \in \pcs} \cost(\conf)$$
    with
    $$\cost(\conf) = \riskg(\mathcal{I}(\datasettrain, \conf) \mid \datasetval)$$
    
    \item The mapping $\conf \mapsto \riskg(\dots)$ is exactly a black box:
    \begin{itemize}
        \item No closed-form expression
        \item Evaluation requires training / resampling a model
        \item Mixed / hierarchical search space\\
        (numeric / integer / cat / conditional HPs)
    \end{itemize}
    \item $\riskg$ notation here is simplified to the use of a single validation set,
    in practice $\riskg$ might be a resampled performance, e.g., by using CV
\end{itemize}
\end{frame}


\begin{frame}{Why Not Manual Tuning?}
\begin{columns}[onlytextwidth]
    \column{.85\textwidth}
    \begin{itemize}
        \item Relies on expert intuition about the learner and the dataset
        \item Very inefficient and not automated
        \item Poorly reproducible -- choices are hard to document and communicate
        \item Prone to bias: humans revisit familiar settings and miss unusual but good regions
    \end{itemize}
    \vspace{0.3cm}
    \textbf{We want a systematic, adaptive, and reproducible alternative.}
    \column{.15\textwidth}
    \begin{center}
        {\Huge\emoji{control-knobs}}\\[0.4cm]
        {\Huge\emoji{thinking-face}}\\[0.4cm]
    \end{center}
\end{columns}
\end{frame}


\begin{frame}[t]{Why Not Just Grid or Random Search?}
  \small
  \begin{itemize}
    \item Simple, parallelizable, but \textbf{non-adaptive}: every config is sampled independently of previous results $\leadsto$ no exploitation of structure, budget wasted on obviously unpromising regions
    \item Scales poorly in higher-dim.\ search spaces
    \item Expensive objectives demand an optimizer that \textbf{learns from its own evaluations} and follows promising local directions
  \end{itemize}
  \vspace{0.1cm}
  \begin{center}
    \includegraphics[width=0.6\textwidth]{images/bo_vs_rs_branin.pdf}\\
    \footnotesize Yellow $\star$ = global minima. RS scatters uniformly, BO concentrates around minima
  \end{center}
\end{frame}

\begin{frame}{The Core Idea of Bayesian Optimization}
\begin{itemize}
    \item Fit a cheap \textbf{surrogate model} $\surro$ to the observations collected so far
    \item Use the surrogate to form an \textbf{acquisition function} that trades off
    \begin{itemize}
        \item \textbf{Exploitation:} sample where $\surro$ predicts good values
        \item \textbf{Exploration:} sample where $\surro$ is uncertain
    \end{itemize}
    \item Evaluate the true objective only at the most \textbf{informative} point, update the surrogate, repeat
\end{itemize}
\vspace{0.3cm}
\begin{center}
\textbf{Sequential, model-based, sample-efficient} -- designed for exactly the setting where each evaluation is valuable.
\end{center}
\end{frame}


\begin{frame}{BO as an AutoML Backbone}
\begin{itemize}
    \item BO is the \textbf{backbone} of many modern AutoML systems (e.g., SMAC, Spearmint, BoTorch, Optuna)~\liturl{Snoek et al. 2012}{https://arxiv.org/abs/1206.2944}
    \item It is typically much more \textbf{sample-efficient} than random or evolutionary search on HPO benchmarks
    \item It provides a principled way to handle
    \begin{itemize}
        \item Noisy evaluations
        \item Multiple objectives (accuracy vs.\ runtime, vs.\ fairness, \ldots)
        \item Multi-fidelity
        \item Warm-starting
        \item Priors
    \end{itemize}
\end{itemize}
\end{frame}

\begin{frame}{Historical Roots: DACE and Expensive Simulations}
BO did not originate in ML -- it grew out of the \textbf{design and analysis of computer experiments} (DACE) community~\liturl{Sacks et al. 1989}{https://projecteuclid.org/journals/statistical-science/volume-4/issue-4/Design-and-Analysis-of-Computer-Experiments/10.1214/ss/1177012413.full}.
\vspace{0.2cm}
\begin{itemize}
    \item Classical engineering setting: a single run of a physics / CFD / finite-element simulator can take hours or days
    \item Idea: treat the simulator as an unknown function, fit a \textbf{Gaussian process} (aka Kriging) to a small design, and use it as a cheap proxy
    \item \textbf{EGO} (efficient global optimization,~\liturl{Jones et al. 1998}{https://link.springer.com/article/10.1023/A:1008306431147}) pairs a GP with \textbf{expected improvement} -- the direct ancestor of today's BO
\end{itemize}
\vspace{0.2cm}
Typical applications: aerodynamic shape, car crash / airfoil simulations, reservoir / groundwater models, climate / weather code calibration, chip / circuit design.
\end{frame}
